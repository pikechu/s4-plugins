# Campaign Completion 插件设计规格

## 1. 目标与范围

为《The Settlers IV History Edition》3.15（`S4_Main.exe` 2.50.1516.0）和 Settlers United 开发 32 位 ASI 插件，在单人固定地图及战役任务选择界面显示已通关标识，并可靠记录玩家正常获胜。

自动记录仅在以下条件全部得到可靠确认时发生：

```text
本地玩家正式获胜
AND 当前游戏不是多人游戏
AND 当前地图不是随机生成地图
AND 当前地图具有稳定身份
```

插件不得修改游戏存档、官方战役进度、资源包、Settlers United 文件或其他插件。开发测试期间，仅允许增删或替换本项目生成的 `CampaignCompletion*.asi` 和 `CampaignCompletion` 配置目录。游戏及 Settlers United 启动时产生的正常日志、配置和缓存写入不属于插件主动修改。

## 2. 已确认的运行环境

- 游戏与现有插件均为 PE32/i386。
- `S4_Main.exe` 文件版本为 2.50.1516.0。
- 本地 `S4ModApi.dll` 版本为 2.3.2.0。
- Settlers United、SettlersUnitedLogger、S4HD-Patch 和 S4WarriorsLib 均通过 `S4ModApi.dll!S4CreateInterface` 接入。
- S4ModApi 提供菜单页面枚举、UI 帧回调、GUI 元素绘制回调、地图初始化回调、鼠标回调和自定义 UI。
- S4ModApi 未提供可直接使用的胜利事件、当前地图文件名、多人状态或随机地图状态接口。
- PDB 中存在 `CStateVictoryScreen`、战役页面状态类、`CMapFile`、`CRandomMaps`、`CStateMDRandomMapParameters` 和 `CStateLobbyMultiplayerType` 等候选符号。

所有候选符号仍需在阶段 1 的完整报告和阶段 2 的运行时诊断中验证；本规格不把符号名称本身视为已验证行为。

## 3. 技术路线

采用“混合式、符号优先”方案：

1. 使用 S4ModApi 承担其已经支持的页面识别、GUI 观察、绘制、鼠标和地图初始化能力。
2. 使用 `S4_MainR.pdb` 确认 S4ModApi 缺失的胜利、地图身份、多人和随机地图候选函数或结构。
3. 仅为缺失能力安装最少量 Hook。
4. Hook 目标必须同时经过游戏版本、模块范围、函数签名和必要的结构不变量校验。
5. 任何关键数据无法确认时安全降级为只写诊断日志，不写通关记录。

纯粹依赖菜单文字、胜利画面或固定坐标的推断不作为自动记录依据。深度 Hook 完整菜单状态机仅作为个别页面无法适配时的最后补充。

## 4. 组件架构

### 4.1 PluginRuntime

- 管理延迟初始化、安全禁用和卸载。
- `DllMain` 只保存模块句柄并触发轻量延迟初始化，不读取 JSON、不安装 Hook、不等待线程。
- 检查 EXE、S4ModApi、Settlers United 和候选签名。
- 关键校验失败时进入“仅日志、安全禁用”状态。

### 4.2 GameStateProbe

聚合 S4ModApi、PDB 定位函数和最少量 Hook 的结果，生成不可变 `GameStateSnapshot`。快照至少包含：

- 会话 ID 和采样时间
- 当前页面与页面来源
- 地图路径、内部 ID 和显示名候选
- 本地玩家 ID 与参与状态
- 游戏是否正式结束
- 本地玩家是否获胜
- 是否多人游戏
- 是否随机生成地图
- 每个字段的数据来源和可信度

### 4.3 CompletionDetector

- 只消费 `GameStateSnapshot`，不直接读游戏内存。
- 按会话状态机判断是否产生通关事件。
- 使用会话 ID 和稳定地图 ID 防止重复事件。
- 默认保留首次通关时间，重复获胜不重复写文件。

### 4.4 MapIdentityResolver

按以下优先级解析稳定身份：

1. 规范化后的地图完整相对路径
2. 地图文件名与已确认目录上下文
3. 游戏内部唯一 ID
4. `campaign:<campaign_id>:mission:<mission_id>`
5. 已验证的页面 ID 与控件 ID 映射

显示文字只能作为日志和人工核对信息，不能作为唯一 ID。路径统一使用相对游戏目录的反斜杠形式，并进行不区分大小写的规范化。

### 4.5 CompletionStore

- 在内存中维护不可变完成记录快照，供渲染线程读取。
- 工作线程负责配置、JSON、备份和日志落盘。
- 提供添加、删除、重新加载和查询接口。
- 所有 JSON 写入使用临时文件、复读校验、备份和原子替换。

### 4.6 MenuOverlay

- 从 S4ModApi GUI 元素绘制回调采集实时页面和控件数据。
- 调用页面适配器把控件映射为 `MissionEntry`。
- 只对具有可靠稳定 ID 且已完成的条目绘制覆盖层。
- 覆盖层不创建点击区域、不拦截输入、不改变键盘导航。

## 5. 会话状态机与可信度

### 5.1 数据流

```text
菜单选择候选地图
→ 地图初始化
→ 建立新会话和身份快照
→ 观察游戏状态
→ 捕获正式结束或胜利候选
→ 重新采样本地玩家、模式和地图身份
→ 完成可信度门控
→ 投递异步持久化事件
```

菜单选择仅为候选身份。地图初始化后必须重新读取游戏内部信息，以避免选择项与实际载入内容不一致。载入存档会创建新会话；只有载入后发生正式胜利才允许记录。

### 5.2 可信度等级

- `Confirmed`：来自已通过运行时测试的内部字段或明确函数。
- `Corroborated`：两个相互独立且已验证的信号一致。
- `Unknown`：无法读取、结构校验失败或来源尚未验证。

自动记录要求本地参与、正式结束、非多人、非随机和稳定地图身份均为 `Confirmed`。本地玩家获胜必须为 `Confirmed`，或由两个独立胜利信号达到 `Corroborated`。任一关键字段为 `Unknown` 时只记录诊断信息。

### 5.3 必须排除的情况

- 玩家失败或主动退出
- 单纯返回主菜单
- 载入获胜前的存档但没有再次获胜
- 仅显示简报而未正式获胜
- 多人游戏、观察者模式或 AI 获胜
- 随机生成地图
- 游戏崩溃或强制结束
- 缺少稳定地图身份

## 6. 菜单识别与页面适配

统一接口为：

```cpp
class ICampaignPageAdapter {
public:
    virtual ~ICampaignPageAdapter() = default;
    virtual bool Matches(const MenuSnapshot& page) const = 0;
    virtual std::vector<MissionEntry>
        EnumerateEntries(const MenuSnapshot& page) const = 0;
};
```

`MenuSnapshot` 包含页面枚举、内部状态类、控件 ID、实时矩形、纹理、内部关联值、隐藏/禁用/悬停/选中状态和辅助文字。

`MissionEntry` 至少包含：

- `stable_id`
- `campaign_id`
- `mission_id`
- `map_file`
- `display_name`
- `x`、`y`、`width`、`height`
- `enabled`
- `selected`
- 身份解析来源与可信度

适配顺序：

1. Single Player 固定地图列表
2. The Dark Tribe
3. The Three Races
4. New World
5. Trojans
6. Great Crusades
7. 其他可识别战役页面

页面适配器只能保留页面特有的匹配和映射规则；控件采集、稳定 ID 解析、选择约束和绘制由共享组件实现。无法建立可靠映射的页面不绘制，只输出控件快照。

## 7. 覆盖层表现与快捷键

首版实现 `CornerCheck`：

- 在条目实时矩形右上角绘制绿色勾。
- 位置由实时矩形、配置偏移和缩放计算，不保存绝对屏幕坐标。
- 使用实际绘制表面和柱状留白参数适配分辨率及 HD Patch。
- 不覆盖主要文字，不接管鼠标。

快捷键：

- `Ctrl+Shift+C`：标记当前唯一选中条目为已完成。
- `Ctrl+Shift+U`：取消当前唯一选中条目的完成状态。
- `Ctrl+Shift+R`：请求工作线程重新加载配置和 JSON。

手动修改不要求胜利状态，但要求页面适配器返回且仅返回一个 `selected=true`、稳定 ID 可靠的条目。零个或多个选中项均拒绝操作并写日志。

## 8. 数据格式与写入安全

`completed_maps.json` 顶层包含格式版本和完成记录数组。每条记录保存稳定 ID、地图路径、战役/任务 ID、辅助显示名、完成时间、记录来源、游戏版本和插件版本。

写入事务：

1. 在内存副本中应用修改。
2. 写入 `completed_maps.json.tmp`。
3. 刷新文件缓冲区。
4. 复读并校验临时 JSON 的语法、版本、必需字段和稳定 ID 唯一性。
5. 将有效原文件复制为 `completed_maps.json.bak`。
6. 使用 Windows 原子替换操作提交临时文件。
7. 提交失败时保留原文件并记录错误。

主 JSON 损坏时禁止任何覆盖操作。若备份有效，可用只读方式载入备份并明确告警，但必须等待人工修复或明确恢复操作后才能重新启用写入。

## 9. 线程与 Hook 安全

- 游戏和渲染回调只更新轻量状态或投递有界事件。
- JSON、配置解析和常规日志 I/O 在工作线程进行。
- 不在每帧扫描内存或读取磁盘。
- Hook 必须通过 S4ModApi 共享入口优先注册；只有 API 缺失时才使用本插件 Hook。
- Hook 安装采用事务方式。部分失败时撤销本插件本次已安装的 Hook。
- 卸载顺序为：停止接收事件、移除监听器和 Hook、限时清空写队列、释放资源。
- 插件不 Hook `ddraw.dll`，避免与 Settlers United、HD Patch 和 ExtraZoom 争用渲染代理入口。

## 10. 故障降级

- S4ModApi 不可用：安全禁用插件。
- 游戏版本或函数签名不匹配：禁用对应探针；关键探针缺失时禁止自动记录。
- 映射数据库损坏：禁用相关页面适配器，不影响游戏。
- 覆盖层异常：禁用覆盖层，不影响完成记录模块。
- 数据文件异常：切换为只读数据模式，不覆盖原文件。
- 未处理的插件内部异常不得传播到游戏主循环；异常边界记录上下文后禁用相关子系统。

## 11. 分阶段交付与验证

### 阶段 1：调查报告

- 固定 S4ModApi 源码提交。
- 分析本地 DLL 导出、PDB 符号和现有插件接入方式。
- 列出候选函数、结构、Hook 点、证据和风险。

### 阶段 2：诊断插件

- 构建 Win32 `CampaignCompletionDebug.asi`。
- 只启用运行时、状态探针和日志。
- 记录页面、地图、胜负、多人和随机状态，不写完成 JSON、不绘制勾。

### 阶段 3：自动记录

- 实现固定单人胜利记录、排除规则、JSON 事务和去重。
- 通过失败、退出、存档、随机、多人及重复胜利负面测试后才默认启用。

### 阶段 4：固定地图列表标识

- 验证列表滚动、选择、语言、分辨率和 HD Patch 下的动态矩形。

### 阶段 5：战役页面适配

- 按既定顺序逐页适配和验证。
- 每个页面保留控件枚举日志和实际截图或用户观察结果。

### 阶段 6：手动修正与配置

- 实现快捷键、热重载、备份恢复和错误提示。

### 阶段 7：Release

- 生成 Win32 Release ASI、默认配置、数据库、双语 README、许可证、源码、Visual Studio 工程及 ZIP。
- 检查 PE 架构、导入依赖、包内容和卸载行为。

## 12. 最低发布门槛

- 游戏可正常启动、退出并反复进入菜单。
- 固定单人胜利只记录一次。
- 失败、退出、多人和随机地图均不记录。
- JSON 损坏不会被覆盖。
- 禁用或删除本插件后游戏恢复原行为。
- Single Player 固定地图列表及至少一个战役页面完成实际勾选验证。
- 未经过实际游戏验证的页面必须在 README 中标为实验性或未验证。

## 13. 测试协作

开发者提供逐步测试操作和预期日志。用户协助进入指定菜单、运行固定/随机/多人场景并触发胜利或失败，再反馈现象和日志。测试报告必须明确区分：

- 静态确认
- 本机运行时确认
- 用户操作确认
- 尚未验证

不得把静态推断或未执行测试描述为可靠功能。
